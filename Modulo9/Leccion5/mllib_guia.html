01

Componentes clave e integración con Spark

Lo primero que hay que entender de MLlib es que no vive solo: es parte del ecosistema de Apache Spark. Eso cambia bastante cómo uno lo piensa y lo usa.

Componentes principales

🗂️
DataFrame

La base de todo. MLlib no opera sobre listas o arrays, sino sobre DataFrames distribuidos de Spark SQL.

🔄
Transformer

Transforma datos: normalizar, tokenizar, convertir texto a vectores numéricos.

🧠
Estimator

Los algoritmos de ML. Reciben datos, aprenden, y devuelven un Model entrenado.

📦
Model

El resultado de entrenar un Estimator. Se usa para hacer predicciones sobre datos nuevos.

⛓️
Pipeline

Encadena todo el flujo: preparación → entrenamiento → predicción, como una sola unidad ejecutable.

¿Cómo se integra con Spark?

Una idea que suele pasarse por alto: MLlib no es una librería externa que se conecta a Spark. Es parte de Spark. Eso implica:

Consecuencia práctica Escala automáticamente en clústeres, usa el mismo motor de ejecución (Catalyst + Tungsten) y opera sobre datos distribuidos sin requerir código adicional de paralelismo.

Entrenar un modelo en MLlib se parece más a escribir un .groupBy() que a llamar un model.fit() de scikit-learn. El contexto de ejecución ya está resuelto.

02

DenseVector vs SparseVector

Parece un detalle técnico, pero en realidad es una decisión de memoria y rendimiento con impacto real a escala.

Diferencia conceptual

Tipo ¿Qué almacena? Representación interna Ejemplo
DenseVector Todos los valores, incluyendo ceros Array completo [1.0, 0.0, 0.0, 5.0]
SparseVector Solo los valores distintos de cero Índices + valores (4, [0, 3], [1.0, 5.0])

¿Cuándo usar cada uno?

DenseVector
  • Pocos datos o pocas dimensiones
  • Baja proporción de ceros
  • Variables numéricas continuas (edad, ingreso, temperatura)
  • Cuando la simplicidad importa más que la memoria
SparseVector
  • Alta dimensionalidad con muchos ceros
  • Datos textuales (TF-IDF, bag of words)
  • One-hot encoding de categorías
  • Sistemas de recomendación (usuarios × ítems)
Ejemplo real Un corpus con 10.000 palabras únicas, donde cada documento usa ~20. Guardar 9.980 ceros por documento es ineficiente. SparseVector almacena solo los índices y valores de esas 20 palabras relevantes: misma información, fracción del espacio.
03

Algoritmos en MLlib

MLlib no pretende competir con scikit-learn en variedad de algoritmos. Su apuesta es diferente: lo necesario para producción a escala.

Supervisados
Logistic Regression
Clasificación binaria / multiclase
→ Churn: ¿el cliente se va o no?
Random Forest
Ensamble de árboles de decisión
→ Scoring y ranking de clientes
Linear Regression
Predicción de valores continuos
→ Precio de propiedades
GBTClassifier
Gradient Boosted Trees
→ Detección de fraude
No supervisados
K-Means
Clustering por centroides
→ Segmentación de clientes
ALS
Filtrado colaborativo
→ Recomendación (Netflix-style)
LDA
Latent Dirichlet Allocation
→ Modelado de tópicos en texto
Bisecting K-Means
Clustering jerárquico divisivo
→ Categorización de documentos
Perspectiva clave MLlib no compite con scikit-learn en variedad. Compite en volumen: los mismos algoritmos corriendo sobre terabytes de datos distribuidos, sin cambiar la API.
04

Pipeline: del notebook a producción

El Pipeline es probablemente el concepto más subestimado de MLlib. No es solo una forma de ordenar código; es la diferencia entre un experimento y algo que funciona en producción.

Estructura de un pipeline típico

Paso 1
StringIndexer
Categorías → índices numéricos
Paso 2
VectorAssembler
Columnas → vector único de features
Paso 3
StandardScaler
Normaliza la escala de las variables
Paso 4
LogisticRegression
Entrena el modelo final

Sin Pipeline vs Con Pipeline

✗ Sin Pipeline
⚠️ Código disperso, difícil de seguir
⚠️ Transformaciones manuales en cada predicción
⚠️ Inconsistencias entre entrenamiento e inferencia
⚠️ Difícil de versionar y reproducir
✓ Con Pipeline
✅ Flujo lineal y legible
✅ Una sola llamada .transform() en producción
✅ Garantiza que los datos pasan por los mismos pasos
✅ Serializable: se guarda y recarga completo

Código de referencia

from pyspark.ml import Pipeline
from pyspark.ml.feature import StringIndexer, VectorAssembler, StandardScaler
from pyspark.ml.classification import LogisticRegression

# 1. Definir pasos del pipeline
indexer   = StringIndexer(inputCol="categoria", outputCol="categoria_idx")
assembler = VectorAssembler(inputCols=["edad", "ingreso", "categoria_idx"],
                              outputCol="features")
scaler    = StandardScaler(inputCol="features", outputCol="features_scaled")
lr        = LogisticRegression(featuresCol="features_scaled", labelCol="churn")

# 2. Ensamblar y entrenar
pipeline       = Pipeline(stages=[indexer, assembler, scaler, lr])
pipeline_model = pipeline.fit(train_df)

# 3. Predecir (aplica todos los pasos automáticamente)
predicciones = pipeline_model.transform(test_df)

# 4. Guardar para producción
pipeline_model.save("/modelos/churn_pipeline_v1")
Reflexión clave El pipeline no es solo orden. Es lo que permite pasar de "tengo un modelo en un notebook" a "tengo algo que puedo ejecutar en producción sin romper todo". La serialización del pipeline garantiza que los datos de inferencia pasan exactamente por los mismos pasos que los de entrenamiento.

Resumen conceptual

Cuando estas piezas se acomodan en la cabeza, MLlib deja de parecer complejo:

DataFrame

La base: datos distribuidos como punto de partida y llegada.

Vectores

La representación: Dense para datos densos, Sparse para alta dimensionalidad.

Estimators

Los algoritmos: aprenden de datos y producen modelos.

Pipeline

El orquestador: garantiza reproducibilidad de extremo a extremo.

Cambio de perspectiva Una vez que entiendes estas piezas, ya no estás simplemente entrenando modelos. Estás diseñando procesos de datos escalables y reproducibles. Esa es la diferencia real entre MLlib y una librería de ML convencional.
